# 악보 분석 기반 재생·자동 페이지 넘김·스코어 팔로잉 계획

작성일: 2026-06-13
대상 버전: v0.2.x+
상태: 📋 계획 수립 (구현 미착수)

---

## 0. 한 줄 요약

PDF 악보를 **기하학적으로 분석**(시스템·마디·보표 bbox)한 결과와 Sibelius **MusicXML**(음악 의미)을 **마디 인덱스로 정렬**하면, 앱에서 재생·자동 페이지 넘김·메트로놈·악보 분해/결합·(궁극적으로) 마이크 기반 스코어 팔로잉을 단계적으로 구현할 수 있다.

- **Phase 0(먼저)** = 합주 모드 **동기 페이지 넘김** — 지휘자가 "넘길 목표 시각(T)" 을 함께 보내 전원이 T 에 동시에 넘김 (현재의 "수신 즉시 넘김" 오차 제거).
- **Phase 1** = 템포 기반 **자동 페이지 넘김 + 메트로놈/마디 하이라이트**, 단 **합주/단독 모드를 구분**.

## 1. 배경: 오늘(2026-06-13) 확보한 것

Sibelius → "Microsoft Print to PDF" 로 출력된 **순수 벡터 악보**(`data/Moldau0607.pdf`, 5파트 앙상블: 진호·예진·하진·예완·은석)를 대상으로 분석 파이프라인을 구축했다. (모두 `data/` 에 스크립트·산출물로 존재, **앱 미통합 상태**.)

### 1-1. PDF 기하 분석 (`data/segment_score.py`)
- 콘텐츠 스트림을 `mutool` 로 해석, **CTM 스택 추적** 후 채움 패스를 분류
  - 오선 = 얇은 가로 사각형(보표당 5개)
  - 마디선 = 얇은 세로 사각형 중 ① 끝점이 오선 상·하단에 정확히 정렬, ② 5개 보표 전부에 같은 x, ③ 끝점에 노트헤드 없음(stem 제외)
- 결과: **26 시스템 / 81 마디** 검출, `data/Moldau0607_layout.json` 출력
  - 시스템·**보표별 y-밴드**·마디 bbox (top-down PDF 포인트 좌표)
- ⚠️ **핵심 교훈**: PDF 좌표계는 y축이 아래→위(+). 이미지 row 로 변환할 때 `row = (page_h − y_pdf)` 로 뒤집어야 함 (안 그러면 시스템 상하가 뒤바뀜).
- ✅ 검출한 마디 번호(1, 5, 8, 11 … 77)가 Sibelius 가 인쇄한 번호와 **완벽히 일치** → 마디 인덱스 정렬의 신뢰 기반.

### 1-2. 파트 분해 (`data/build_parts.py`)
- 각 연주자 보표 밴드를 모든 시스템에서 크롭 → 연주자별 파트보 PDF (`data/parts/`).
- WYSIWYG(원본 그대로) 방식. 200 DPI 래스터, 마디 번호·제목 헤더 포함.

### 1-3. OMR (광학 악보 인식) PoC
- 음고(`data/pitch_poc.py`): 노트헤드(채움 타원 w≈6 h≈5) y위치 → 높은음자리표 + G장조(F#) 매핑. **신뢰도 높음**(검증 완료).
- 쉼표(`data/rest_poc.py`): 글리프 모양·세로 위치로 온/4분/8분/16분/점쉼표 분류. **정확**.
- 음길이(빔/깃발 검출): 마디 검산(합=6/8) 기준 **약 50%** — 빔이 slant 되거나 16분/8분 혼동에서 오차.

### 1-4. 음원 합성 (`data/multi_synth2.py` + `data/dur_detect.py`)
- 5파트 전사 → 스테레오 MP3 (numpy 합성 + `pip install imageio-ffmpeg` 의 정적 ffmpeg 인코딩).
- **핵심 타이밍 규칙(사용자 지정)**: 모든 마디는 **동일한 시간 슬롯**을 차지. 마디 안에서 음표·쉼표 길이를 **순차 배치**(마디 앞·사이 멈춤 제거) 후 **합을 정확히 6 eighth 로 스케일** → 마디 길이 균일, 템포 균일, 박자선이 5파트 정렬.
- 산출: `data/Moldau_ensemble_5parts_halfspeed.mp3` (2배 느림), `data/Moldau_Yejin_melody.mp3`.

## 2. 핵심 아이디어: PDF 기하 ⊕ MusicXML 하이브리드

두 데이터는 **상호보완적**이며, **마디 인덱스**가 둘을 잇는 linchpin 이다.

| | PDF 기하 분석 (확보) | MusicXML (Sibelius export) |
|---|---|---|
| 담당 | **어디에** 있는가 — 시스템·마디·보표 페이지 좌표 | **무엇을** 연주하는가 — 음고·음길이·파트·반복·템포·박자표 |
| 강점 | 앱이 이미 띄우는 그 PDF 위에서 직접 동작 (하이라이트/크롭/페이지 넘김) | 정확한 연주, 조옮김, 깔끔한 재조판 |
| 한계 | 음악 의미 약함(빔 ~50%) | 페이지의 실제 픽셀 레이아웃 없음 |

**정렬 맵**: `MusicXML 마디 N ↔ PDF 마디 N bbox`. 우리 마디 번호가 인쇄 번호와 일치하므로 1:1 매핑이 신뢰성 있게 성립.

이 매핑이 풀어주는 것:
- 재생 위치(마디 인덱스) → 그 마디의 페이지·bbox 조회 → **자동 페이지 넘김 + 하이라이트**
- 마디 단위 점프(시각적)
- 파트 분해/결합 (크롭 = 원본 그대로 / MusicXML 재조판 = 깔끔)

⚠️ **주의**: 반복(`:||`)·도돌이·멀티마디 쉼표가 있으면 Sibelius 마디 번호와 bbox 매핑이 어긋날 수 있어 정렬 로직 필요. (예: `data/Moldau0607.pdf` page13 끝 = `:||` + 빈 마디 2개.)

## 3. 단계별 계획

### 🚦 Phase 0 — 합주 모드 동기 페이지 넘김 (타임스탬프 예약 방식)

**목표**: 기존 합주 모드 페이지 넘김의 **시점 오차 제거**. 현재는 신호 수신 즉시 넘기므로, 전달 지연·기기 렌더 딜레이 때문에 기기마다 넘어가는 순간이 어긋난다.

**아이디어(사용자 제안)**: 지휘자가 페이지 넘김 신호를 보낼 때 **"몇 초 후 넘긴다"는 목표 시각 T 를 함께** 전달 → 모든 기기가 신호를 미리 받아 두고 **정해진 T 에 동시에 실행**. 신호 전달과 실제 넘김 사이에 시간 여유를 둬서 전달 속도·기기 딜레이 편차를 흡수.

**현재 흐름**: `conductor showPage → broadcastCollaborationPageChange(page) → follower 수신 즉시 showPage`. → 전달 지연/지터/렌더 딜레이로 불일치.

**바꿀 흐름**:
1. **시계 동기화 서브프로토콜** (신규, 핵심) — WebSocket 으로 ping-pong(Cristian/SNTP 식)을 주고받아 각 연주자의 **지휘자 대비 시계 offset + RTT** 추정. 연결 시 1회 + 주기적 갱신(드리프트 보정).
2. **예약 메시지** — 지휘자: `T = now + lead`(예 1.5~2초) 계산 → `broadcast {page, turn_at: T(지휘자 시계)}`.
3. **로컬 예약** — 연주자: 측정한 offset 으로 `T_local = elapsedRealtime_now + (T − 지휘자_now)` 환산 → 그 시각에 `showPage` 예약(Handler/coroutine delay). **로컬 스케줄링은 단조 시계(elapsedRealtime)** 사용, 공유 기준은 offset 환산.
4. **지휘자 자신도 T 에 넘김** — 자기 화면도 동일 시각에 → 전원 동기.
5. **폴백** — T 가 이미 지난 채 수신(lead 부족/심한 지연)되면 즉시 넘김 + 로그.

**lead time 트레이드오프**: 길수록 지연·지터에 견고하나 응답성↓. 수동 넘김은 지휘자가 살짝 미리 눌러야 함. Phase 1 의 템포 기반에서는 **페이지 끝 시점을 미리 알므로 자연스럽게 예약** 가능(응답성 문제 없음).

**재사용/수정 지점**: `broadcastCollaborationPageChange()` 메시지에 `turn_at` 필드 추가, 수신부에 예약 스케줄러 추가, 연결 핸드셰이크에 시계 동기 ping-pong 추가.

**예상 난이도**: ⭐⭐⭐ (시계 동기화가 핵심, 페이지 넘김 로직 자체는 기존 재사용)

**이 Phase 0 가 Phase 1 합주 모드의 페이지 넘김 메커니즘이 된다.**

### 🥇 Phase 1 — 템포 기반 자동 페이지 넘김 + 메트로놈/마디 하이라이트 (MVP)

**목표**: 마이크 없이, 설정된(또는 MusicXML) 템포로 악보를 자동으로 넘기며, 메트로놈 비트와 현재 마디 하이라이트를 표시.

**왜 1단계인가**: 음고·마이크 같은 어려운 요소 없이, 오늘 만든 `layout.json` + 앱의 기존 페이지 동기화 경로만으로 구현 가능. 가치가 크고 위험이 낮음.

**기존 코드 재사용 (핵심 발견)**:
- `PdfViewerActivity.showPage(index)` (약 522행) = 페이지 표시의 중심.
- 그 안에서 이미 `broadcastCollaborationPageChange(index)` (약 587행) 호출 → **지휘자→연주자 페이지 동기화 경로가 이미 존재**.
- ⇒ 자동 넘김 = **원격 지휘자 대신 "템포 클록"이 `showPage()` 를 호출**하는 것. 새 메커니즘 거의 불필요.

**⭐ 모드 구분 (합주 vs 단독) — 반드시 분기**:

| | 단독 모드 | 합주 모드 — 지휘자 | 합주 모드 — 연주자(follower) |
|---|---|---|---|
| 페이지 넘김 | 로컬 템포 클록이 직접 `showPage` | 템포 클록이 결정 → **Phase 0 예약 신호**로 전원 동시 넘김 | **자동 넘김 안 함**. 지휘자 예약 신호로만 넘김 |
| 메트로놈/하이라이트 | 로컬 클록 기준 표시 | 로컬 클록 기준 표시 + 위치·템포 **브로드캐스트** | **로컬 표시하되 지휘자 브로드캐스트(템포·현재 마디)에 동기** |

- 핵심 원칙: **합주 모드에서 "넘기는 권한"은 지휘자에게만**. 연주자는 메트로놈·마디 하이라이트만 로컬로 보여주고, 페이지는 지휘자가 준 **Phase 0 예약 타임스탬프**에 맞춰 넘긴다.
- 연주자 메트로놈 동기화: 지휘자가 **현재 마디 인덱스 + 템포(BPM) + 비트 위상 기준시각**을 주기적으로 브로드캐스트 → 연주자는 자기 시계(offset 환산)로 비트를 재생성. (Phase 0 의 시계 동기 인프라 재사용.)
- 단독 모드는 네트워크·예약 없이 로컬 클록만으로 전부 구동(가장 단순).

**구현 항목**:
1. **마디 레이아웃 리소스** — `Moldau0607_layout.json`(및 향후 파일별) 을 앱 에셋/DB 로 탑재. 마디→(페이지, bbox) 조회 제공.
2. **템포 클록** — coroutine 기반. 박자표(MusicXML 또는 수동 입력)·BPM 으로 마디·비트 진행.
   - 마디 인덱스 증가 → 페이지 P 조회 → P 가 현재와 다르면 `showPage(P)`.
3. **오버레이 View** — PDF ImageView 위에 그림:
   - 현재 마디 bbox 사각형(반투명) 또는 진행 커서
   - 비트마다 플래시/점멸 (메트로놈 시각화), 선택적 클릭음(SoundPool — 이미 효과음 시스템 보유)
4. **재생 컨트롤** — 시작/정지/템포 조절(D-pad/터치), 시작 마디 지정.

**기술 난관 — 좌표 변환 (PDF pt → 화면 px)**:
- `layout.json` 좌표는 **PDF 포인트(595×842)** 기준.
- 앱은 `renderPageAtSinglePageTarget` / `renderPageAtTwoPageTarget` / `calculateOptimalScale` 에서 **oversample → downscale + 상하 클리핑 + 중앙 여백 + 두 페이지 모드**로 렌더.
- 오버레이는 **동일한 스케일·오프셋·클리핑**을 거쳐 화면 픽셀로 변환해야 함. 두 페이지 모드(좌/우 페이지 결합)·클리핑 케이스를 반드시 처리.
- ✅ 권장: 렌더러가 사용하는 변환 파라미터(scale, offsetX/Y, clipTop/Bottom, 페이지 배치)를 한 곳에서 노출해 오버레이가 재사용.

**예상 난이도**: ⭐⭐⭐ (좌표 변환이 대부분의 일)

### 🥉 Phase 3 — 마이크 기반 박자/온셋 인식 자동 넘김

**목표**: 연주를 마이크로 들으며 **템포·비트를 검출**해, 고정 템포가 아니라 **실제 연주에 적응**하는 자동 넘김 + 라이브 메트로놈.

**핵심 통찰**: **온셋·박자·템포 검출은 음고 추정보다 훨씬 쉽고, 다성(합주)에서도 동작**한다. (음 시작과 박 간격은 에너지/스펙트럼 변화만으로 잡힘.)

**얻는 것 (쉽고 견고)**:
- 온셋 검출(스펙트럼 플럭스) → 음 시작 시점
- 템포 추정(온셋 간격 자기상관/히스토그램) → BPM
- 비트 위상 추적 → 메트로놈, 현재 마디 비트 하이라이트

**솔직한 한계 — 박자 ≠ 절대 위치**:
- 박자 검출은 "지금 템포·방금 박" 은 주지만 "지금 17마디" 는 못 줌.
- **비트 카운팅**(시작점부터 박 수 세기): 간단하나 음 놓침/추가 시 **드리프트 누적**.
- **리듬 지문 매칭**(음고 없이 온셋 패턴을 MusicXML 기대 리듬에 DTW): 더 견고하나, **이 곡처럼 16분음표 마디가 반복되면 마디 구분 모호**(음고 필요).

**현실적 스위트스팟**:
- 마이크로 **템포·비트 검출** → 메트로놈 + 마디 비트 하이라이트
- 박자표로 마디당 박 수를 알고 비트 카운트 → 페이지 경계에서 넘김
- 드리프트는 **싼 재정렬**로: 페이지 넘김 시 위치를 "이 페이지 첫 마디"로 리셋, 또는 연주자가 탭/D-pad 로 보정. 페이지 넘김은 몇 마디에 한 번뿐이라 작은 오차는 대부분 무해.

**Android 실현 경로**:
- `AudioRecord`(PCM) → 단시간 FFT → 스펙트럼 플럭스 온셋 → 템포 (태블릿 실시간 충분).
- **TarsosDSP** (Android 호환 Java DSP): 온셋·비트/템포·YIN 음고 내장 → 바닥부터 구현 불필요.
- 선행 검증: 합성 WAV/실연주 녹음에 **Python 으로 온셋+템포 프로토타입** → 검증 후 TarsosDSP 포팅.

**예상 난이도**: ⭐⭐⭐⭐

## 4. 공통 기술 토대 (Phase 1~4 공유)

1. **마디 ↔ bbox 정렬 맵**
   - PDF 기하 분석(확보) + MusicXML 마디. 반복/빈마디 정렬 로직 포함.
   - 앱이 소비할 JSON 스키마 정의 필요: `{measure_index, page, bbox(px@pt), system, beats}`.
2. **좌표 변환 유틸 (PDF pt → 화면 px)**
   - 렌더러의 scale/offset/clip/two-page 배치를 단일 함수로 노출.
   - Phase 1 오버레이, 향후 하이라이트/주석(027 로드맵)과 공유.
3. **기존 합주 동기화 인프라 재사용**
   - `showPage()` + `broadcastCollaborationPageChange()` = 자동 넘김의 실행 경로.
   - 합주 모드에서 "지휘자 = 재생 엔진" 으로 두면, 연주자 화면이 마디 단위로 정확히 따라옴.
4. **MusicXML 파서**
   - 마디·박자표·템포·파트·(반복) 추출. (Android 측 경량 파서 또는 사전 변환 JSON.)

## 5. 위험 / 한계 정리

- **시계 동기 정확도 / 드리프트** (Phase 0): offset 추정 오차·시계 드리프트가 동기 정밀도를 좌우. 주기적 재동기 + 충분한 lead time 으로 흡수. lead 가 짧으면 일부 기기가 T 를 놓쳐 폴백(즉시 넘김)으로 빠짐.
- **좌표 변환 정확도** (Phase 1 최대 난관): 두 페이지 모드·상하 클리핑·중앙 여백까지 일관되게 매핑해야 오버레이가 어긋나지 않음.
- **반복/도돌이 정렬**: MusicXML 마디 번호 ↔ PDF bbox 어긋남 처리.
- **OMR 음길이 정확도 ~50%**: PDF-only 리듬은 부정확 → 정확한 리듬·연주는 **MusicXML 필수**. (PDF 분석은 "위치", MusicXML 은 "음악".)
- **마이크 위치 추적 드리프트**(Phase 3): 절대 위치는 음고 없이는 근본적으로 모호 → 재정렬/보정 UX 설계 필요.
- **다성 음고 분리**(Phase 4 합주): 현재 기술로는 비현실적, 단성부터.

## 6. 권장 진행 순서

1. **Phase 0** — 합주 모드 동기 페이지 넘김(시계 동기 + 타임스탬프 예약). ← **여기서 시작**. 기존 합주 기능을 즉시 개선하고, Phase 1 합주 모드의 토대가 됨.
2. **마디 ↔ MusicXML 정렬 맵** 확정 (공통 토대) — Sibelius 에서 MusicXML 1개 export → 정렬 JSON 생성.
3. **Phase 1** — 템포 클록 자동 넘김 + 메트로놈/마디 하이라이트 오버레이 (단독/합주 모드 분기, 합주는 Phase 0 예약 신호 사용).
4. **Phase 2** — 태블릿 플레이버 분리.
5. **Phase 3** — 마이크 온셋/템포 검출(Python PoC → TarsosDSP).
6. **Phase 4** — 단성 음고 팔로잉 → (장기) 합주.
