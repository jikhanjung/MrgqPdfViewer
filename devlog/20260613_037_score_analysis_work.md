## 0. 개요

Sibelius 로 작성된 PDF 악보 1개를 분석해 **시스템·마디 단위로 분해**하고, 나아가 **연주자별 파트보 추출 · 음표/쉼표 인식(OMR) · 음원 합성(MP3)** 까지 PoC 로 수행했다. 모든 작업물은 `data/` 디렉토리에 스크립트와 결과물로 보존되어 있으며, **앱 코드에는 아직 통합하지 않았다**.

핵심 결과:
- **26 시스템 / 81 마디** 자동 검출, 검출 마디 번호가 Sibelius 인쇄 번호와 **완전 일치**.
- 5개 연주자 파트보 PDF 생성.
- 음고·쉼표 인식 신뢰성 확인, 5파트 합주 MP3 합성.

## 1. 대상 파일

- `data/Moldau0607.pdf` — 스메타나 "Die Moldau (Vltava)", 5인 앙상블(진호·예진·하진·예완·은석), 6/8박자, G장조, 13페이지.
- 생성 경로: **Sibelius → "Microsoft Print to PDF"**. 즉 순수 **벡터** PDF (래스터 이미지 없음, 모든 음표·오선·기호가 채움 패스).
- 페이지: A4(595.32×841.92pt), 페이지당 2시스템 × 5보표.

---

## 2. 작업 1 — PDF 구조 분석 (시스템 · 마디 검출)

### 2-1. 방법 (`data/segment_score.py`)
1. `mutool clean -d` 로 콘텐츠 스트림 압축 해제 → `mutool show` 로 오브젝트별 스트림 덤프.
2. 콘텐츠 스트림을 직접 해석하며 **CTM(현재 변환 행렬) 스택 추적**:
   - 베이스 변환 `0.75 0 0 -0.75 0 841.92 cm` (Y반전·0.75배), 이후 `q`/`Q` push/pop, 로컬 `cm` 합성.
   - 각 채움(`f`)마다 패스 점들의 **디바이스 좌표 바운딩 박스** + 곡선 포함 여부 기록.
3. 채움 패스 분류:
   - **오선** = 높이<1.5pt, 너비>80pt 가로 사각형 → 5개씩 묶어 보표.
   - **마디선** = 너비<2pt 세로 사각형 중 ① 끝점이 보표 상·하단 오선에 ±1.5pt 정렬, ② 5개 보표 전부에 같은 x, ③ 끝점에 노트헤드 없음(stem 배제).
4. 보표를 세로 간격으로 시스템 그룹화, 시스템을 **y 내림차순(페이지 위쪽 먼저)** 정렬.

### 2-2. 핵심 버그와 해결: Y축 방향 ⚠️
- 파싱 좌표는 **PDF 사용자 공간(y=0 이 페이지 하단)**. 이걸 이미지 row 로 그대로 쓰면 위/아래 시스템이 뒤집혀 **두 시스템의 마디선이 서로 바뀜**.
- 해결: `image_row = (page_height − y_pdf)` 로 뒤집고, 시스템을 y_pdf 내림차순 정렬.
- 오버레이(`data/Moldau0607_p1_overlay.png`)로 빨강(윗 시스템)·파랑(아래 시스템) 마디선이 실제와 정확히 일치함을 확인.

### 2-3. stem 오검출 대응
- 16분음표가 빽빽한 마디(예: page 6)에서 음표 기둥(stem)이 여러 보표에 정렬돼 마디선으로 오검출 → 12마디로 과검출.
- 사용자 힌트 반영: **"마디선 끝점은 오선 상·하단에 정확히 정렬되고, stem 은 반드시 노트헤드에 붙는다."**
  - 끝점 정렬 ±1.5pt 로 엄격화, 5개 보표 전부 정렬 요구, **끝점에 노트헤드 글리프가 있으면 stem 으로 제외**.
  - 인접 이중선(간격<15pt) 병합.
- 결과: page 6 이 정상 3+3 마디로 수렴.

### 2-4. 결과
- **26 시스템 / 81 마디**.
- 검출 마디 번호(1, 5, 8, 11 … 44, 47 … 74, 77)가 Sibelius 가 인쇄한 번호와 **완전 일치** (육안 대조 검증). page1 첫 시스템만 4마디, 마지막(page13 하단)은 5마디(음악 3 + 반복줄 `:||` + 빈 마디 2), 나머지 24개 시스템은 모두 3마디.
- 출력: `data/Moldau0607_layout.json` — 시스템·**보표별 y-밴드**·마디 bbox (top-down PDF 포인트).

## 3. 작업 2 — 연주자별 파트보 추출 (`data/build_parts.py`)

- 각 연주자 보표의 y-밴드를 26개 시스템 전부에서 크롭(상하 ±23pt 여백 포함) → A4 페이지에 한 줄씩 재조판.
- 헤더(제목·작곡가·연주자 이름, 한글 Noto CJK), 각 줄 좌측에 **시작 마디 번호**(파란색).
- 200 DPI 래스터. WYSIWYG(원본 그대로) 방식.
- 출력: `data/parts/Moldau_{1..5}_{Jinho,Yejin,Hajin,Yewan,Eunseok}.pdf` (각 3페이지).
- 한계: 래스터 방식이라 확대 한계, 일부 시스템에서 인접 보표 내용이 가장자리에 희미하게 비침.

## 4. 작업 3 — 음표/쉼표 인식 (OMR PoC)

### 4-1. 음고 (`data/pitch_poc.py`)
- 노트헤드 = 곡선 채움 + **w≈6, h≈5** 타원. 크기 signature 로 깔끔하게 분리.
- y위치 → 높은음자리표 라인/스페이스 → 음이름. 조표 G장조 → F#.
- 예진 멜로디 38개 음표 검출, 라벨이 모든 노트헤드에 정확히 일치(`data/Moldau0607_pitch_demo.png`). **신뢰도 높음.**

### 4-2. 쉼표 (`data/rest_poc.py`)
- 글리프 모양·크기·세로 위치로 분류:
  - 온/마디쉼표 = 채움 사각형(w≈6,h≈2.4), 4번째 선 아래 / 2분쉼표 = 가운데 선 위
  - 4분쉼표 = w≈5,h≈15 곡선 중앙 / 8분쉼표 = w≈5.6,h≈9 / 16분쉼표 = h≈13(갈고리 2개) / 점 = 우측 작은 점
- 진호 보표 검증: m1·m2·m4 = 점4분쉼표+8분쉼표+8분쉼표, m3 = 온쉼표 → 6/8 정확 일치(`data/Moldau0607_rest_demo.png`). **정확.**

### 4-3. 음길이 (빔/깃발, `data/dur_detect.py`)
- 빔 = 비곡선 가로 막대(w>9). 기둥에 걸린 빔 개수: 2줄=16분, 1줄=8분. 깃발=8분, 없으면 4분. 부점 ×1.5.
- 마디 검산(합=6/8 eighth) 기준 **약 50%** 정확 (진호 50/예진 49/하진 45/예완 74/은석 38%).
- 한계: slant 된 빔, 16분/8분 혼동에서 오차. → PDF-only 리듬은 부정확, 정밀 리듬엔 MusicXML 필요.

## 5. 작업 4 — 음원 합성 (전사 → MP3)

### 5-1. 도구 부트스트랩
- WSL2 기본 환경에 pip/ffmpeg/numpy 없음 → `python3 -m ensurepip` 후 `pip install imageio-ffmpeg`(정적 ffmpeg 바이너리 포함, sudo 불필요) + numpy.

### 5-2. 타이밍 모델 (사용자 피드백으로 2회 개선)
- **v1(비례-x)**: 음표 x좌표를 그대로 시간으로 환산 → 마디 앞·사이 멈춤 + 템포 출렁임 발생.
- **v2(균등 마디, `data/multi_synth2.py`)**: 사용자 지정 — **모든 마디가 동일한 시간 슬롯**. 마디 안에서 음표·쉼표 길이를 순차 배치(멈춤 제거) 후 합을 **정확히 6 eighth 로 스케일**. → 마디 길이 균일, 템포 균일, 박자선이 5파트 정렬. 쉼표는 소리 없는 시간 진행으로 반영.

### 5-3. 합성
- numpy 가산합성(배음 + 어택/감쇠 엔벨로프 + 미세 비브라토 + Schroeder 리버브), 파트별 pan/음색.
- ffmpeg(libmp3lame)로 MP3 인코딩.
- 출력:
  - `data/Moldau_Yejin_melody.mp3` — 예진 멜로디 단독 (81마디, 405음).
  - `data/Moldau_ensemble_5parts_halfspeed.mp3` — 5파트 합주, 2배 느림 (총 2,058음, 2분 39초).

## 6. 핵심 교훈

1. **PDF y축은 아래→위(+)**. 이미지/화면 좌표로 변환할 때 반드시 뒤집어야 함. (향후 앱 오버레이 좌표 변환에서도 동일 주의.)
2. **마디선 vs stem 구분**: 끝점이 오선 상·하단에 정확 정렬 + 노트헤드 미부착 = 마디선. (사용자 힌트가 그대로 판별 규칙이 됨.)
3. **검출 마디 번호 == 인쇄 마디 번호** → MusicXML 마디 인덱스와의 정렬 신뢰 기반 확보.
4. **PDF 분석 = "위치", MusicXML = "음악"**. OMR 음길이 ~50% 한계로, 정확한 연주·리듬은 MusicXML 이 필요. 둘은 상호보완적.
5. 합주 음원 타이밍은 **마디 길이 균등 + 마디 내 스케일**이 핵심 (멈춤·출렁임 제거).
